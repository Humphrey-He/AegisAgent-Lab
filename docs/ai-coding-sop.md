# AI Coding SOP

## 目标

建立一套适合 3-5 年后端开发者使用的 AI Coding 工作流，使 AI 能稳定参与需求拆解、代码阅读、测试补齐、Bug 排查、日志分析和 PR Review，同时保证过程可验证、结果可复盘、风险可控制。

本 SOP 面向 Go、C#、Rust 后端项目。所有 AI 输出都必须经过人工判断和工程验证后才能进入主干。

## 任务准入标准

### 优先使用 AI 的任务

- 边界清晰的小功能开发，例如新增一个 CRUD 接口、补齐一个服务方法、扩展一个数据转换流程。
- 已有模式明确的增量开发，例如参照现有 Handler、Service、Repository、DTO、测试风格补齐同类实现。
- 单元测试、表格驱动测试、边界条件测试、错误路径测试的补充。
- 错误日志、堆栈、失败测试、线上告警的初步分析。
- 模块职责、调用链、接口契约、依赖关系的代码阅读。
- PR 或 diff 的初步审查，重点发现行为错误、测试缺口、安全风险和回归风险。
- 变更说明、接口说明、实验记录、复盘报告的整理。

### 谨慎使用 AI 的任务

- 跨多个核心模块的大范围重构。
- 权限、支付、资金、密钥、隐私数据、数据删除等高风险功能。
- 需求规则不清晰、业务上下文依赖大量隐性知识的任务。
- 缺少自动化测试保护且改动会影响核心链路的任务。
- 涉及并发、事务一致性、幂等、分布式状态迁移的复杂改动。

### 不应直接交给 AI 独立完成的任务

- 未经人工确认的生产数据修复。
- 未经安全审查的鉴权、授权、加密、签名、密钥管理改动。
- 未经架构评审的跨服务协议变更。
- 需要业务负责人确认规则含义的需求决策。
- 任何无法通过测试、日志、审查或手工验收证明正确性的改动。

## 上下文收集清单

在发起 AI Coding 前，先收集并裁剪上下文。目标是让 AI 看到足够材料，但不被无关信息干扰。

### 必备上下文

- 任务背景：为什么要做、当前问题是什么、成功后系统表现如何。
- 技术栈：Go、C# 或 Rust，以及框架、测试框架、数据库、消息队列等关键组件。
- 目标范围：允许修改的模块、文件、接口、测试。
- 明确约束：不能改变的行为、兼容性要求、性能要求、安全要求、代码风格要求。
- 验收方式：需要运行的测试、需要观察的日志、需要验证的接口或边界场景。

### 代码上下文

- 相关目录结构。
- 入口代码，例如 Controller、Handler、Endpoint、Command、Job。
- 业务代码，例如 Service、UseCase、Domain、Repository。
- 类型定义，例如 DTO、Request、Response、Entity、Error 类型。
- 已有相似实现，用于让 AI 复用本项目风格。
- 已有测试文件，用于对齐命名、断言、fixture、mock 方式。
- 构建和测试命令。

### 问题上下文

- 复现步骤或触发条件。
- 错误日志、堆栈、请求 ID、关键指标。
- 预期行为与实际行为。
- 最近相关变更。
- 已排除的原因和人工判断。

### 上下文裁剪原则

- 优先提供与任务直接相关的最小闭环：入口、业务逻辑、依赖接口、测试。
- 对大型文件只提供相关函数、类型和调用方。
- 对日志只保留时间、错误、堆栈、请求关键信息和上下游关联信息。
- 对不可公开的信息先脱敏，例如 token、手机号、邮箱、客户名称、内部域名。

## AI Coding 执行流程

### 1. 任务规格化

在让 AI 修改任何内容前，先写清楚以下信息：

- 背景：当前问题或需求来源。
- 目标：完成后用户或系统可观察到的变化。
- 范围：允许 AI 阅读和修改的文件或模块。
- 约束：不能改变的接口、行为、数据格式、错误语义。
- 验收：测试命令、检查项、人工验证步骤。

推荐使用 `prompts/requirement-breakdown.md` 先拆解任务，再决定是否进入实现。

### 2. 代码阅读与方案确认

先让 AI 阅读代码并输出理解，而不是直接改动：

1. 使用 `prompts/code-reading.md` 阅读相关模块。
2. 要求 AI 说明模块职责、调用链、关键类型、外部依赖和潜在风险。
3. 要求 AI 给出最小改动方案和测试计划。
4. 人工检查方案是否符合任务范围和已有架构。

方案不清楚时，先补充上下文或缩小任务，不进入实现。

### 3. 小步执行

执行时遵循小步闭环：

1. 只让 AI 修改完成当前目标所需的最小文件集合。
2. 每轮修改后查看 diff，确认没有无关格式化、重命名、依赖升级或大范围重构。
3. 运行与改动相关的最小测试集。
4. 根据测试失败、编译错误、Review 反馈继续迭代。
5. 任务完成后运行更高一层的回归测试或集成检查。

### 4. Bug 排查流程

Bug 类任务先定位再修复：

1. 收集现象、复现步骤、日志、堆栈、预期行为、最近变更。
2. 使用 `prompts/bug-investigation.md` 要求 AI 输出可能原因、证据和验证动作。
3. 人工确认最可能根因。
4. 让 AI 做最小修复，并补充能复现问题的回归测试。
5. 验证新测试能覆盖问题，原有测试继续通过。

### 5. 测试生成流程

测试类任务必须先确认测试意图：

1. 列出正常路径、边界条件、错误路径、兼容性场景。
2. 使用 `prompts/test-generation.md` 生成测试草案。
3. 检查测试是否只是覆盖实现细节，必要时改为验证外部行为。
4. 运行测试并修正不稳定测试。
5. 记录覆盖到的风险和未覆盖原因。

### 6. PR Review 流程

Review 类任务先关注风险，再关注风格：

1. 提供 PR 目标、diff、相关测试结果和风险背景。
2. 使用 `prompts/pr-review.md` 要求 AI 按严重程度输出发现。
3. 人工核实每个发现是否有证据。
4. 将结论分为必须修改、建议修改、无需处理。
5. 对误报记录原因，用于优化后续提示词。

## 验收清单

AI Coding 结果进入完成状态前，必须逐项检查：

- 任务目标已完成，且没有扩大需求范围。
- diff 只包含与任务相关的文件和必要改动。
- 没有引入未解释的新依赖、新框架或新运行时要求。
- 没有绕过鉴权、校验、错误处理、日志、事务或并发保护。
- 错误路径有明确处理，错误语义符合项目已有风格。
- 新增或修改逻辑具备相应测试，至少覆盖正常路径和关键失败路径。
- 相关测试、构建或静态检查已运行，并记录结果。
- 敏感信息没有出现在代码、日志、Prompt、报告或提交信息中。
- 对外接口、数据结构、配置项或兼容性行为如有变化，已写入说明。
- AI 输出中的假设已经被人工确认或在记录中标明为未采用。

## 实验记录规范

每次 AI Coding 实验都应沉淀到报告或案例记录中，便于后续比较人工基线、AI 辅助收益和质量风险。

### 记录字段

```text
任务名称：
语言/项目：
任务类型：
任务背景：
输入材料：
使用 Prompt：
AI 工具/模型：
人工基线耗时：
AI 辅助耗时：
Prompt 轮次：
AI 产出摘要：
人工修改次数：
运行的验证命令：
验证结果：
Review 问题：
质量评分：
效率结论：
风险与教训：
后续改进：
```

### 质量评分建议

- 正确性：是否满足需求、是否通过测试、是否处理边界和错误路径。
- 范围控制：是否避免无关改动、格式化、重构和依赖变化。
- 可维护性：是否符合项目风格、命名、分层、异常处理和测试习惯。
- 可解释性：是否能说明方案、权衡、风险和验证方式。
- 效率收益：是否减少人工阅读、编码、测试或 Review 时间。

### 报告沉淀要求

- 单次实验记录写入 `reports/ai-coding-practice-report.md`。
- 人工基线数据写入 `reports/baseline-report.md`。
- 可复用的 Prompt 改进入库到 `prompts/`。
- 可复用的评估任务入库到 `eval-cases/`。
- 每次报告都要写清楚样本数量、结论边界和不适用场景。
