# Case 005: PR Review

## 任务背景

评估 AI 是否能对后端 diff 做有效审查，优先发现行为错误、回归风险、测试缺口、安全问题和性能问题。该案例适合中小型 PR、自包含 bugfix、接口改动、测试补齐和局部重构审查。

评估重点是发现质量和证据质量，而不是输出大量泛泛建议。

## 输入材料

- 技术栈说明：Go、C#、Rust 及框架、测试框架。
- PR 目标：需求背景、修复问题或重构目的。
- 变更范围：涉及模块、接口、数据结构、配置。
- Git diff：完整或聚焦后的 diff。
- 测试结果：已运行命令、失败输出、未运行原因。
- 业务约束：兼容性、错误码、权限、数据一致性要求。
- 风险关注：并发、事务、鉴权、性能、幂等、日志。

## 操作步骤

1. 使用 `prompts/pr-review.md` 输入 PR 目标、约束、测试结果和 diff。
2. 要求 AI 按严重程度输出发现，并为每个发现提供证据。
3. 人工核实每个发现，标记为真问题、误报或低价值建议。
4. 对真问题记录严重程度、修复建议和是否补测试。
5. 对误报记录误判原因，用于优化 Prompt。
6. 对比人工 Review 发现，统计 AI 命中率和漏报项。
7. 记录是否建议合入以及残余风险。

## 评分维度

| 维度 | 分值 | 评分说明 |
| --- | ---: | --- |
| 行为问题发现 | 25 | 能发现可能导致错误结果、异常或回归的改动。 |
| 测试缺口识别 | 20 | 能指出缺少的关键测试和风险覆盖。 |
| 安全稳定性 | 15 | 能识别鉴权、输入校验、并发、事务、性能等风险。 |
| 证据质量 | 15 | 每个发现有明确 diff、测试或约束依据。 |
| 噪音控制 | 15 | 少输出泛泛建议，误报率可控。 |
| 建议可执行性 | 10 | 修复建议具体、范围合理、便于开发者处理。 |
