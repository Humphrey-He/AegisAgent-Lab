# PR Review Prompt

## 使用场景

用于让 AI 对后端 PR、commit diff 或局部变更做初步审查。适合在人工 Review 前发现行为错误、回归风险、测试缺口、安全问题和性能问题，不用于替代最终人工审核。

## 输入字段

- 技术栈：Go、C#、Rust 及相关框架。
- PR 目标：本次变更要解决的需求或问题。
- 变更范围：涉及模块、文件、接口、数据结构。
- Diff：待审查的代码差异。
- 测试结果：已运行的测试、失败信息、未运行原因。
- 风险关注：鉴权、数据一致性、并发、性能、兼容性等重点。
- 业务约束：不能改变的行为、错误码、数据格式、接口契约。

## 模板

```text
你是一个严格但务实的后端代码审查助手。请审查以下 diff，优先发现可能导致线上问题的风险。没有证据的问题不要输出为结论。

技术栈：
[填写 Go/C#/Rust、框架、测试框架]

PR 目标：
[填写变更目的]

变更范围：
[填写涉及模块、文件、接口、数据结构]

业务约束：
[填写不能改变的行为、错误码、兼容性要求]

风险关注：
[填写鉴权、数据一致性、并发、事务、性能、安全等]

测试结果：
[填写已运行测试、失败信息、未运行原因]

Diff：
[粘贴 diff]
```

## 输出格式

```text
1. 总体结论
   - 是否建议合入：
   - 主要风险：

2. 必须修改
   - [严重程度] 文件/位置：
     问题：
     证据：
     影响：
     建议：

3. 建议修改
   - 文件/位置：
     问题：
     证据：
     建议：

4. 测试缺口
   - 缺口：
   - 建议补充：

5. 安全与稳定性检查
   - 鉴权/授权：
   - 输入校验：
   - 错误处理：
   - 并发/事务：
   - 性能：

6. 无问题说明
   - 如果没有明显问题，请说明已检查的维度和残余风险。
```

## 质量检查点

- 发现必须有 diff 或测试结果作为证据。
- 优先输出行为错误、回归风险、安全问题、测试缺口。
- 不输出泛泛的代码风格建议，除非会影响可维护性或正确性。
- 必须区分必须修改和建议修改。
- 如果没有发现问题，要说明检查过的范围和剩余风险。
