# 测试生成 Prompt

## 使用场景

用于为已有后端函数、服务、接口或 Bug 修复补充测试。适合生成 Go `testing` 表格驱动测试、C# xUnit/NUnit 测试、Rust `cargo test` 单元测试和集成测试草案。

## 输入字段

- 技术栈：语言、测试框架、mock 工具、断言库。
- 被测目标：函数、服务、接口、模块职责。
- 被测代码：核心实现、依赖接口、错误类型。
- 业务规则：正常路径、边界条件、错误路径。
- 已有测试风格：命名、fixture、mock、断言方式。
- 测试范围：只补单元测试，或包含集成测试、回归测试。
- 已知风险：历史 Bug、并发、事务、时间、随机数、外部依赖。

## 模板

```text
请为以下后端代码生成测试。测试应验证外部行为，不要只锁定实现细节。请优先复用已有测试风格。

技术栈和测试框架：
[填写 Go testing / xUnit / NUnit / cargo test，以及 mock/断言库]

被测目标：
[填写函数、服务、接口或模块职责]

被测代码：
[粘贴核心代码、依赖接口、错误类型]

业务规则：
[填写正常路径、边界条件、错误路径]

已有测试风格：
[粘贴相似测试、fixture、mock 示例]

测试范围：
[填写单元测试、集成测试、回归测试]

已知风险：
[填写历史 Bug、并发、事务、时间、随机数、外部依赖等]
```

## 质量检查点

- 必须覆盖正常路径、关键边界条件和错误路径。
- 测试命名应表达业务行为，而不是内部实现细节。
- 生成的测试应能稳定运行，避免依赖真实时间、随机数和外部服务。
- mock 和 fixture 应符合项目已有测试风格。
- 必须说明未覆盖风险，不能假装测试已经覆盖全部场景。
